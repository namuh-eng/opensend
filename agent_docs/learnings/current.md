# Current OpenSend Learnings

This file compacted the previous one-file-per-learning inbox into a single current reference. Prefer live code and tests over historical notes when they disagree; the source index at the bottom preserves provenance for the compacted entries.

## Durable rules

### Architecture and service boundaries

- Keep Next.js route handlers as adapters: auth/session/API-key resolution, request/query validation, response mapping, and audit/metrics live at the edge; business rules belong in `packages/core` services or shared handler modules.
- Thin-adapter extractions are proven by route/service unit tests and, for auth or tenant-sensitive work, real Playwright coverage rather than route mocks.
- Control-plane `services/api` is still a Hono skeleton/adapter surface; public API compatibility remains primarily in Next routes until explicit thin-adapter PRs move behavior.
- When a root compatibility path collides with a dashboard page (`/emails`, `/broadcasts`, `/api-keys`, etc.), use method-specific middleware rewrites or explicit root route files and document only paths backed by runtime evidence.

### Auth, tenant isolation, and dashboard sessions

- Tenant-owned reads/writes must resolve a concrete `userId`; API keys without an owner should fail closed, and dashboard flows must use Better Auth sessions/cookies instead of localStorage API-key assumptions unless the UI intentionally calls an API-key-only route.
- Dashboard create/update flows for domains, contacts, templates, webhooks, audience metadata, and billing should exercise the same production routes as API clients, with dashboard-session support added at the adapter boundary when needed.
- Tenant predicates must cover secondary/detail/actions too: cancel, attachments, logs, contact detail, preview, replay, exports, and relationship routes can be missed even after CRUD looks scoped.
- Permission checks should go through shared helpers such as API-key permission gates rather than scattered string comparisons.

### Email send, idempotency, quota, and webhooks

- Send routes persist intent first, reserve quota in the same transaction, then enqueue provider work; queue/SQS and workers own provider retries, while provider retry state copied onto email rows is for visibility.
- Single and batch idempotency preserve the public replay contract, limit replay lookup to the current 24-hour window, and persist enough original response metadata to return the same envelope.
- Webhook deliveries use the `email_events`/`webhook_deliveries` pipeline with Svix-compatible `svix-*` headers and HMAC signing; add event types only when current lifecycle transitions can emit them.
- Lifecycle/contact/domain events may reuse nullable `email_events.email_id`; do not create parallel event systems without a real boundary reason.

### SES, domains, DNS, receiving, and deliverability

- Domain verification and provider state are region-aware and domain-routed, not transparent failover. Cache keys must include SES identity and region, and verification updates must invalidate all relevant dashboard/provider caches.
- SES identity setup should be idempotent: adopt existing identities when possible, treat MAIL FROM setup as non-fatal where intended, and emit concrete DKIM/SPF/DMARC/return-path/tracking records from one source of truth.
- Inbound receiving belongs behind the standalone ingester: provider ingress, storage, forwarding/reply processing, and dashboard read APIs are separate readiness layers.
- For SNS topic allowlists, converting fallback chains into unions can silently widen trust boundaries; legacy cross-purpose fallbacks must be included only when no specific topic is configured.
- Deliverability dashboards must use provider feedback events (`email_events`) rather than queue/provider-send statuses as proxies for bounce/complaint/open metrics.

### Billing, pricing, quotas, and plans

- Hosted pricing is one combined OpenSend Cloud plan family for API sends plus broadcast fanout, with discrete monthly Stripe prices instead of yearly toggles or continuous sliders.
- Billing/public plan DTOs are route-specific; do not assume `/api/billing/plans` and summary envelopes expose identical shapes.
- Quota enforcement reserves usage with conditional database updates before insert/send side effects; dashboard usage and docs should treat billing surfaces as the active source of truth.

### Automations, broadcasts, audience, templates, and exports

- Automation runner slices advance one due step per tick; wait/condition/cancel behavior stores state in `automation_runs.step_states` and additive metadata rather than ad-hoc side tables unless the model demands it.
- Audience/contact/segment/topic/property routes mix public API-key and dashboard-session callers; be explicit about which detail routes are API-key-only and which collection routes accept dashboard sessions.
- Dashboard exports should be bounded, tenant-scoped, redacted where needed, and durable only when the feature requires it; immediate CSV slices should cap row counts and use shared dashboard-session export boundaries.
- Template rendering preserves the legacy stored-template path while React Email support is selected by registry metadata and preview endpoints remain ID-scoped.

### Docs, SDKs, API compatibility, and product surface

- Public docs are first-party OpenSend docs, backed by implemented routes/features. Keep `public/docs/**/*.md` indexed through `/docs/llms.txt`, and run `bun run docs:generate` when public docs change.
- Human docs routes and raw markdown are separate surfaces: markdown remains the LLM corpus, while pretty `/docs` routes render the product docs shell.
- SDK/package slices live in repo-local packages with deterministic test harnesses and shared DTO contracts where possible; edge/runtime docs should use fetch examples rather than unsupported SDK imports.
- Public OpenAPI/root-alias docs must be backed by middleware rewrites or App Router route files, not parity aspirations.

### Testing and validation

- Auth, tenant, persistence, and security work needs real-dependency Playwright proof with Better Auth/session/API-key fixtures; browser route mocks are not backend correctness proof.
- Keep unit tests at the pure service/helper boundary and route tests at adapter behavior. For new `services/api` routes, update all `createApp()` mocks because static route registration imports shared handlers.
- Avoid time-sensitive fixtures that expire; scheduled-email tests should generate future timestamps relative to the test run.
- When Vitest/Bun workspace aliases misresolve deep package paths, prefer explicit test aliases or package entrypoints instead of weakening tests.

### Operations, deployment, migrations, and observability

- Docker/ingester builds need `bun install --ignore-scripts` where postinstall files are absent; deployment scripts under ignored `scripts/` must be intentionally tracked or generated elsewhere.
- Next middleware cannot import Node-only clients unless configured for Node middleware; otherwise Edge runtime failures can break every request.
- Deploying ECS services that rely on new secrets or Container Insights requires verifying task definitions, image digests, cluster settings, and real alarm datapoints before calling the rollout healthy.
- Drizzle SQL and meta snapshots can drift independently; migration parity gates should compare migrate-provisioned and push-provisioned schemas and treat full drift as a release blocker.
- Structured JSON logs/CloudWatch EMF and tracing should carry enough request/job context to correlate API, queue, worker, and ingester paths without leaking secrets.
