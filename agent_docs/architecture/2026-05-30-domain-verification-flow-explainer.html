Why your domain stays "Pending" — the actual story

You asked: "what's the problem we're trying to solve?". This page walks through it. There are two bugs, not one. We shipped a fix for the first. The second is still broken — and now we can see it.

1. The happy-path flow (what should happen)

When you add a domain to OpenSend, here's the sequence that should turn it Verified within ~60 seconds of DNS propagation:

1.You add a domain in dashboard→
2.OpenSend tells AWS SES to track it→
3.SES generates DNS records you need to add→
4.You add records at your DNS provider→
5.Status flips to "Verified"

The interesting part is step 5. SES doesn't push you a notification when it has verified a domain. You have to ask SES, every so often, "is this verified yet?" That's what the scheduler does — it's a small process that pings every 60 seconds.

The verification loop, every 60s

Schedulerruns every 60s
POST /jobs/domain-verify → Ingester
Ingesterdoes the reconcile
Reads pending domains from Postgres
For each: asks SES "is this verified?"
If yes → updates Postgres status to "verified"
Tells Redis to forget the old cached "pending" value
Dashboardnext time you refresh
Reads from Redis → MISS → reads fresh from Postgres → sees "verified"

2. Bug #1 — the cache invalidation hole FIXED in #578

The bug: the ingester DID update Postgres correctly. But it called a "tell Redis to forget" function that was secretly a no-op.

How that happened: the function was passed in as a dependency. The dependency had a default value of async () => {} — an empty function. When the ingester forgot to pass in the real one, it silently used the empty one. Cache never got cleared. Dashboard kept serving stale "Pending".

The fix in #578

  1. Made the "tell Redis to forget" dependency required — you can't construct the service without it. Code that forgets now fails to compile.
  2. Deleted the global "convenience" instance that had the bug baked in.
  3. Added a heartbeat table in Postgres — every time the scheduler runs a job, it writes "I'm alive" into a row.
  4. Added a new endpoint /api/health/scheduler that reads those heartbeats and tells you which jobs aren't firing.

That last point matters for Bug #2.

2. Bug #2 — the scheduler doesn't exist in prod STILL BROKEN

This is the actual root cause of "domain stuck Pending 20+ min" you reported. Even with Bug #1 fixed, nothing has changed unless something is calling /jobs/domain-verify every 60s.

In docker-compose (for self-hosters) there's a tiny container called scheduler that does this. In your AWS production deploy, that container doesn't exist. Your ECS deploy script (scripts/deploy.sh) only deploys two services: the app and the ingester. No scheduler.

3. Why we left Bug #2 unfixed (for now)

Fixing it requires touching your AWS infrastructure (Terraform/CDK or ECS task definitions), not application code. That's a different repo and a different deploy path. The three PRs we just merged were code changes only — they fixed Bug #1 cleanly and exposed Bug #2 so you can see it.

Three ways to fix Bug #2 (pick one)

OptionWhat it isEffort
EventBridge Scheduler rules AWS-native cron. Three rules, each POSTs to events.opensend.namuh.co/jobs/<name> every 60s with the bearer token. ~30 min in Terraform/CDK. Best long-term fit.
ECS scheduler service Add a third ECS service running bun /app/job-scheduler.js. Mirrors how docker-compose works for self-hosters. ~1 hour. Closest to the existing setup.
Sidecar container Add the scheduler as a sidecar container in the ingester ECS task definition. Same task as ingester, separate process inside it. ~20 min. Tightest coupling, smallest infra change.
